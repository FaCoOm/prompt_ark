# 扩展端分类重构计划

## Summary
- 按你的方案把扩展内部分类从单一 `category` 改为 `output_modality + category_type + category_key + classification_confidence`，并把旧 prompt 统一迁移到新分类结构后删除旧 `category` 字段；对外仍只暴露一个“分类”文本。
- 分类来源统一收敛到 taxonomy：系统分类只能来自 `taxonomy.v1.json`，自定义分类只来自用户输入，低置信度用 `pending` 表达待确认。
- 不新增第二次大模型调用，沿用当前“先保存、后异步丰富”的节奏，把现有元数据丰富链改成一次返回标题、标签、模态、taxonomy 分类 key 和置信度。

## Key Changes
### 1. 数据模型与迁移
- 在 prompt 本地模型中新增并持久化：`output_modality`、`category_type`、`category_key`、`classification_confidence`。
- 删除旧 `category` 字段；迁移完成后，新结构成为唯一分类真相。
- 加一个 taxonomy 解析层，提供：
  - `authority_category_id -> label`
  - `label_zh/label_en/alt_labels -> authority_category_id`
  - 当前语言下的系统分类列表
- 做一次幂等迁移：
  - 旧 prompt 保留内容本体（title/content/tags 等），统一按新规则重新分类。
  - 迁移时重跑当前分类流程，回写 `output_modality/category_type/category_key/classification_confidence`。
  - `confidence >= 0.8`：迁成 `category_type='system'`。
  - `confidence < 0.8`：迁成 `category_type='pending'`，并保留推荐的 taxonomy key 作为 `category_key`。
  - 用户后续仍可把 `pending` 确认为 `system`，或改成 `custom`。
- `categoryAutoGenerated` 不再承担新方案里的分类状态语义；迁移完成后可删除或仅作为一次性迁移辅助字段。

### 2. AI 分类与保存流程
- 把现有元数据提取链改成一个统一返回结构，至少包含：
  - `title`
  - `tags`
  - `output_modality`：`text | image | video`
  - `recommended_category_key`：必须是 taxonomy key
  - `confidence`：0-1
- 更新分类 prompt 和 provider schema，禁止模型输出自由文本分类名；系统分类只能输出 taxonomy key。
- 所有入口统一走同一分类流程：
  - 手动新建
  - Smart Convert / AI 转换生成
  - 导入后的本地保存
  - 现有右键快速保存入口
- 保存规则定死：
  - 用户已明确选了系统分类或自定义分类时，保存时直接写 `category_type/category_key`，异步丰富不能覆盖用户选择。
  - 用户没选分类时，先保存内容；异步分类回来后：
    - `confidence >= 0.8`：写 `category_type='system'`
    - `confidence < 0.8`：写 `category_type='pending'`
  - 异步丰富仍可补标题、标签、模态、confidence，但不能把已确认的 `system/custom` 改掉。
- 这轮不新增持久化 `source_type`；沿用现有入口上下文和 `sourceContext.convertMethod` 区分来源即可。

### 3. UI 与交互
- 顶部筛选区改成两层：
  - 固定模态筛选：`全部 / 文本 / 图像 / 视频`
  - 分类筛选：分为“系统分类”和“我的分类”
- 系统分类列表直接来自 taxonomy；自定义分类列表从本地 `category_type='custom'` 的 `category_key` 去重聚合。
- prompt 卡片、列表、搜索、分享预览里仍只显示一个“分类”文本：
  - `system`：taxonomy label
  - `custom`：原样显示
  - `pending`：taxonomy label + “待确认”标记
- 编辑弹窗把当前自由文本 `categoryInput` 改成统一分类控件：
  - 选择系统分类
  - 如果当前是 `pending`，可一键“使用推荐分类”
  - 切换为自定义分类并输入名称
- 不在 UI 中暴露 `category_type`、`taxonomy`、`authority_category_id`。
- 右键重命名只保留给自定义分类；系统分类不可改。批量重命名时只更新 `category_type='custom'` 的 `category_key`。

### 4. 存储与兼容面
- 本地存储、sync slim prompt、WebDAV/Obsidian round-trip 都要带上新字段，避免跨设备和导入导出丢状态。
- 对外兼容继续输出单个 `category` 文本：
  - JSON 导出 / 导入兼容老结构
  - Obsidian frontmatter 继续保留 `category`
  - 分享和 Hub 发布继续传单一 `category` 文本
- `category` 的派生语言跟随扩展当前语言。
- 代码里目前只有发布到 Hub，没有实际“从 Hub fork 到本地”的入口；这轮不新增 Hub 协议或 Hub 逻辑，只把本地分类标准化能力准备好，后续如果补 fork 入口再复用。

## Public API / Interface Changes
- prompt 对象新增：
  - `output_modality?: 'text' | 'image' | 'video'`
  - `category_type?: 'system' | 'custom' | 'pending'`
  - `category_key?: string`
  - `classification_confidence?: number`
- 现有 AI 元数据接口返回结构从自由文本 `category` 改为 taxonomy 约束结构；扩展内部只基于 `category_type/category_key` 渲染展示文本，不再持久化旧 `category` 字段。
- 导入旧数据时接受旧结构 `category` 作为一次性迁移输入；迁移完成后内部只保留新结构 `category_type/category_key/...`。

## Test Plan
- 旧数据迁移：
  - 旧 prompt 会被统一重跑当前分类流程
  - 高置信度正确落 `system`
  - 低置信度正确落 `pending`，并保留推荐 taxonomy key
- 新建 prompt：
  - 用户不选分类时，高置信度自动落 `system`
  - 用户不选分类时，低置信度自动落 `pending`
  - 用户手动选系统/自定义后，异步 enrich 不覆盖
- AI 转换 / 右键保存 / 导入：
  - 都走同一异步分类回写逻辑
  - 标题、标签、模态仍能正常补齐
- UI：
  - 模态筛选生效
  - 系统分类 / 我的分类分组正确
  - `pending` 显示“待确认”
  - 自定义分类可重命名，系统分类不可重命名
- 迁移：
  - 旧 prompt 被重分类并写入新字段
  - 旧 `category` 字段迁移后被删除
- 展示：
  - 扩展语言切换后，系统分类展示文本跟随切换

## Assumptions
- 旧数据迁移规则固定为“统一重跑当前分类流程并写入新字段，然后删除旧 `category` 字段”。
- 分类结果采用“先保存、后异步补分类”，不阻塞保存。
- 系统分类对外文本跟随扩展当前语言。
- 这轮不新增 `source_type` 持久化字段，也不新增 Hub fork 功能。
