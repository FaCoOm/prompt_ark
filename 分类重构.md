
## 一、前提约束

这次只改扩展端，规则固定为：

* tag 逻辑不动
* Hub 逻辑暂时不改
* taxonomy config 作为系统默认分类来源
* 用户不能改 taxonomy 定义
* 用户可以在本地把分类改成：

  * 系统分类
  * 自定义分类
* 低置信度在本地由用户自己确认
* 外部展示还是一个“分类”
* 内部拆成：

  * `category_type`
  * `category_key`

---

## 二、扩展端数据来源

扩展端现在会有 3 类来源：

### 1）用户手动新建

用户自己写标题、内容

### 2）AI 转换生成

系统根据已有内容智能转换成 prompt

### 3）从 Hub fork 下来

共享内容同步到本地

这三类入口后面都统一走同一套分类流程。

---

## 三、扩展端分类规则

### 3.1 系统默认分类来源

来自 `taxonomy.v1.json`

例如：

* `general_productivity`
* `writing_editing`
* `marketing_brand`
* `coding_dev`
* `design_visual`

---

### 3.2 LLM 自动分类只做两件事

1. 判断 `output_modality`
2. 推荐一个系统分类 key

例如返回：

```json
{
  "output_modality": "text",
  "recommended_category_key": "marketing_brand",
  "confidence": 0.76
}
```

注意：

LLM 不生成自定义分类。
自定义分类只来自用户自己输入。

---

## 四、创建 / 保存流程

这个是最核心的。

---

### 场景 A：用户手动新建 Prompt

#### 第一步：用户输入内容

用户填写：

* 标题
* prompt 内容

#### 第二步：系统自动分类

调用分类服务，得到：

* `output_modality`
* `recommended_category_key`
* `confidence`

#### 第三步：系统写入默认值

### 如果 `confidence >= 0.8`

直接写：

```ts
category_type = 'system'
category_key = recommended_category_key
```

比如：

```ts
category_type = 'system'
category_key = 'marketing_brand'
```

#### 第四步：用户保存

这时用户也可以手动改分类：

##### 方案 1：接受系统分类

保持不动

##### 方案 2：换成另一个系统分类

例如改成：

```ts
category_type = 'system'
category_key = 'creative_media'
```

##### 方案 3：改成自定义分类

例如输入：

```ts
category_type = 'custom'
category_key = '小红书爆款'
```

---

### 场景 B：低置信度

如果 `confidence < 0.8`

系统先写：

```ts
category_type = 'pending'
category_key = recommended_category_key
```

例如：

```ts
category_type = 'pending'
category_key = 'creative_media'
```

然后前端显示：

* 分类：创意与媒体
* 状态：待确认

这时用户可以：

#### 选项 1：确认这个推荐

确认后改成：

```ts
category_type = 'system'
category_key = 'creative_media'
```

#### 选项 2：改成别的系统分类

例如：

```ts
category_type = 'system'
category_key = 'marketing_brand'
```

#### 选项 3：不用系统分类，直接新建自定义分类

例如：

```ts
category_type = 'custom'
category_key = '短视频脚本模板'
```

这就是你现在最顺的逻辑。

---

### 场景 C：AI 转换生成

AI 转换生成的 prompt，和手动创建完全一样。

只是 `source_type = 'ai_transformed'`

后面仍然是：

* 自动推荐系统分类
* 高置信度直接系统分类
* 低置信度进入 pending
* 用户自己确认成 system 或 custom

---

### 场景 D：从 Hub fork 到本地

如果是从 Hub 下来的 prompt：

#### 默认继承

* `output_modality`
* Hub 的系统分类

也就是：

```ts
category_type = 'system'
category_key = hub.authority_category_id
```

例如：

```ts
category_type = 'system'
category_key = 'coding_dev'
```

然后用户在本地仍然可以改成：

* 另一个系统分类
* 或自定义分类

因为本地就是个人空间。

---

# 五、扩展端展示逻辑

你强调“外部展示还是一样的”，这里我帮你明确。

## 用户界面仍然只展示一个字段：分类

但渲染规则不同。

### 1）系统分类

如果：

```ts
category_type = 'system'
category_key = 'marketing_brand'
```

显示：

`营销与品牌`

### 2）自定义分类

如果：

```ts
category_type = 'custom'
category_key = '小红书爆款'
```

显示：

`小红书爆款`

### 3）待确认分类

如果：

```ts
category_type = 'pending'
category_key = 'creative_media'
```

显示：

`创意与媒体`
并加一个小标记：

`待确认`

所以对外你仍然只有一个“分类”，只是内部更清楚。

# 六、扩展端筛选逻辑

这个地方很关键。

你现在的筛选应该改成：

## 1）模态筛选

固定：

* 全部
* 文本
* 图像
* 视频

## 2）分类筛选

这里虽然对外还是一个分类，但内部来源分两组：

### 系统分类列表

来自 taxonomy config

### 自定义分类列表

来自本地所有 `category_type = custom` 的 `category_key` 去重聚合

最终 UI 可以这样显示：

* 系统分类

  * 通用与效率
  * 写作与编辑
  * 营销与品牌
* 我的分类

  * 小红书爆款
  * 工作流模板
  * 图像实验

但对用户来说，还是“分类筛选”，只是你做了分段。

---

# 七、扩展端编辑逻辑

编辑 prompt 时，分类区域建议这样：

## 分类模块

一个统一的分类控件，支持三种操作：

### 1）选择系统分类

下拉选择 taxonomy 中的值

### 2）使用推荐分类

如果当前是 pending，显示“使用推荐分类”

### 3）切换为自定义分类

输入自定义分类名

这样就够了。

不要在 UI 上暴露太多内部概念，比如：

* category_type
* authority_category_id
* taxonomy

这些都不要给用户看。

---

# 八、扩展端存储规则

你现在只改扩展端，我建议本地存储直接这样：

## 本地持久化字段

必须存：

* `output_modality`
* `category_type`
* `category_key`
* `classification_confidence`
* `tags`

这就够了。

---

# 九、你现在最适合的状态理解

你已经把“状态”尽量压缩掉了，所以这里不要再做复杂状态机。

你可以简单理解：

## 分类状态由 `category_type` 间接表达

### `system`

表示已经确认使用系统分类

### `custom`

表示已经确认使用自定义分类

### `pending`

表示系统推荐了分类，但用户还没确认

这样很轻。
