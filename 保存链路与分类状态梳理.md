# 保存链路与分类状态梳理

## 目的

这份文档只聚焦两块：

1. 按业务入口梳理所有“保存 / 更新 prompt”的链路与闭环
2. 单独梳理分类状态流转、分类链路和当前实现中的几种状态

本文以当前代码实现为准，不只看设计稿。

---

## 一、统一总线

虽然业务入口很多，但当前实现里大部分链路最后都会汇总到同一条保存总线：

1. 入口侧收集原始内容、标题、tags、分类输入、AI 推荐结果
2. 进入后台消息或直接进入统一 metadata 处理
3. 调用 `buildSavedPromptMetadata()` 统一生成：
   - `title`
   - `tags`
   - `output_modality`
   - `output_modality_locked`
   - `category`
   - `category_type`
   - `category_key`
   - `classification_confidence`
   - `needs_category_review`
   - `ai_category_type`
   - `ai_category_key`
   - `ai_category_confidence`
4. 调用 `PromptStorage.save()` / `PromptStorage.update()` / `PromptStorage.bulkSet()`
5. 本地持久化到 `chrome.storage.local.prompts`
6. 按当前同步后端决定是否同步到 WebDAV / Obsidian
7. 广播 `PROMPTS_UPDATED`
8. Popup 重新渲染分类、模态筛选、卡片列表
9. 如果 metadata 不完整，再触发 `asyncEnrichPrompt()` 做二次补全

### 当前统一核心模块

- `popup.js`
- `background.js`
- `lib/context-menu.js`
- `lib/prompt-metadata.js`
- `lib/taxonomy.js`
- `lib/storage.js`
- `lib/ai/enrich.js`

## 二、按业务入口分类

这里按“用户是从哪里触发保存”的角度分。

### 2.1 手动管理类

| 入口 | 触发方式 | 保存消息 | 首次分类来源 | 是否二次 enrich | 闭环 |
|---|---|---|---|---|---|
| 新建提示词 | Popup 点击“新建” | `SAVE_PROMPT` | 表单输入或启发式 fallback | 是 | 保存后刷新列表、定位新卡片、必要时异步补分类 |
| 编辑已有提示词 | Popup 编辑弹窗 | `UPDATE_PROMPT` | 优先沿用原状态；如果用户改分类则按新输入重算 | 是 | 更新版本历史、刷新卡片、必要时异步补分类 |
| 选择已有自定义分类后保存 | Popup 分类控件选择“我的分类”中的已有项 | `SAVE_PROMPT` / `UPDATE_PROMPT` | 直接使用已有自定义分类 | 理想上否 | 业务上应像系统分类一样直接闭环 |
| 编辑弹窗内确认当前分类 | 分类待确认时点“确认当前分类” | `UPDATE_PROMPT` | 以当前表单分类为准，带 `confirm_category_review=true` | 视情况 | 把待确认状态收口成确认态 |
| 编辑弹窗内应用 AI 推荐分类 | 分类待确认时点“应用推荐分类” | `UPDATE_PROMPT` | 直接把 `ai_category_*` 写回表单后保存 | 视情况 | 把 AI 推荐正式落成系统分类或自定义分类 |
| 翻译后更新 | 列表内翻译 / 编辑弹窗翻译 | `UPDATE_PROMPT` | 理论上应沿用原分类，内容和标题改写 | 是 | 保存后刷新；这条属于“更新已有 prompt”而非新建 |

### 2.2 轻量采集类

| 入口 | 触发方式 | 保存消息 | 首次分类来源 | 是否二次 enrich | 闭环 |
|---|---|---|---|---|---|
| 快速添加 | 选中文本后悬浮工具栏“添加到 Prompt Ark” | `QUICK_ADD_SELECTION` | 启发式标题 + 启发式分类 fallback | 是 | 立即保存原文、toast 提示、Popup 定位到新卡片 |
| 右键快速添加 | 右键菜单“Add to Prompt Ark” | 直接在 `context-menu.js` 内保存 | 启发式标题 + 启发式分类 fallback | 是 | 立即保存原文、右上角 toast、Popup 定位到新卡片 |

说明：

- 这两条是同一业务入口族，都是“先存原文，再异步补 metadata”
- 现在实现上分成了两套代码，一套在 `background.js`，一套在 `lib/context-menu.js`

### 2.3 AI 转换类

| 入口 | 触发方式 | 保存消息 | 首次分类来源 | 是否二次 enrich | 闭环 |
|---|---|---|---|---|---|
| 智能转换 | 选中文本后悬浮工具栏“智能转换” | `SMART_CONVERT_SELECTION` | AI 返回 `recommended_category_type/key/confidence` | 否，默认跳过 | 保存后直接展示结果；低置信度进入待确认 |
| 右键智能转换 | 右键菜单“Smart Convert to Prompt” | 直接在 `context-menu.js` 内保存 | AI 返回 `recommended_category_type/key/confidence` | 否，默认跳过 | 保存后 toast，Popup 定位到新卡片 |
| 抓取上下文 | `Ctrl+Shift+G` 或 `GRAB_CONTEXT` | `SMART_CONVERT_SELECTION` | 与智能转换相同，只是输入文本来自整页抽取 | 否，默认跳过 | 保存为带 `sourceContext` 的智能 Prompt 卡片 |
| 智能转换失败回退 | AI 调用失败时自动回退 | 同上 | 启发式 fallback | 是 | 仍然保存原文，不丢数据 |

说明：

- 这是最“先分类、后保存”的入口族
- AI 会同时给出：
  - `output_modality`
  - `recommended_category_type`
  - `recommended_category_key`
  - `confidence`
  - `tags`
- 高置信度可以直接落确认态
- 低置信度会直接落待确认态

### 2.4 媒体生成类

| 入口 | 触发方式 | 保存消息 | 首次分类来源 | 是否二次 enrich | 闭环 |
|---|---|---|---|---|---|
| 图片转提示词保存 | `image-prompt.html` 独立页点保存 | `SAVE_PROMPT` | 分类仍走统一 fallback，模态显式锁 `image` | 是 | 保存后按钮临时变成成功态 |
| 视频结果直接保存 | Popup 视频结果面板点保存 | `SAVE_PROMPT` | 分类未传结构化字段，最终仍走统一 fallback，模态显式锁 `video` | 是 | 保存后刷新列表 |
| 视频结果转到编辑弹窗再保存 | 视频结果点“编辑后保存” | `SAVE_PROMPT` / `UPDATE_PROMPT` | 先进入手动编辑链路 | 是 | 进入标准手动管理闭环 |

说明：

- 图片和视频链路都显式锁定了 `output_modality`
- 但分类并没有真正“显式确认”
- 所以它们的分类闭环仍依赖统一 metadata 处理和后续 enrich

### 2.5 批量 / 外部进入类

| 入口 | 触发方式 | 保存消息 | 首次分类来源 | 是否二次 enrich | 闭环 |
|---|---|---|---|---|---|
| 导入 JSON / 扫描导入 | Import 面板 | `IMPORT_PROMPTS` | 传入结构化分类则归一化，否则重新推断 | 是 | 全量写入后刷新、定位第一条 |
| Obsidian 回流 | WebDAV / Obsidian merge | `PromptStorage.bulkSet()` | 读取 frontmatter 中的结构化分类 | 否，除非后续人工编辑 | 合并到本地缓存后参与正常展示 |
| 默认 Prompt 种子 | 安装时 seed | `PromptStorage.bulkSet()` | `default-prompts.js` 自带系统分类 | 否 | 直接作为已确认系统分类存在 |

## 三、各入口的统一闭环

不管入口是什么，只要最终落到保存，当前系统的闭环通常包括下面几段：

### 3.1 保存闭环

1. 生成或收集 prompt 数据
2. 统一归一化 metadata
3. 写入 `PromptStorage`
4. 触发 `PROMPTS_UPDATED`
5. Popup 重绘：
   - `loadPrompts()`
   - `renderModalityFilters()`
   - `renderCategories()`
   - `renderPrompts()`
6. 通过 `pendingPromptReveal` 把用户视角定位到新卡片

### 3.2 分类闭环

1. 首次保存时先得到一个初始分类状态
2. 如果 `needs_category_review=false`，该条目可直接视为“分类已闭环”
3. 如果 `needs_category_review=true`：
   - 卡片操作按钮会被禁用
   - 编辑弹窗会显示分类审核面板
   - 用户需要确认当前分类，或应用 AI 推荐，或换成其他分类
4. 对于允许异步 enrich 的入口，还会自动跑一次后台补全：
   - 若 AI 高置信度明确，可能自动把待确认收口
   - 若仍低置信度，则继续保留待确认状态

### 3.3 同步闭环

1. 先写本地 `chrome.storage.local.prompts`
2. 再根据后端配置同步：
   - 本地 only
   - WebDAV
   - Obsidian
3. Obsidian frontmatter 也会带上分类结构字段，保证回流时不丢状态

## 四、分类相关的核心字段

当前实现里，分类不是一个字段，而是一组字段一起工作：

| 字段 | 作用 |
|---|---|
| `category_type` | 当前分类类型，可能是 `system` / `custom` / `pending` |
| `category_key` | 当前分类的真实 key；系统分类是 taxonomy key，自定义分类是用户输入文本 |
| `category` | 展示层文本，不是权威来源 |
| `classification_confidence` | 当前分类结果的置信度 |
| `needs_category_review` | 当前分类是否仍需用户确认 |
| `ai_category_type` | AI 推荐的分类类型 |
| `ai_category_key` | AI 推荐的分类 key |
| `ai_category_confidence` | AI 推荐置信度 |
| `manual_custom_category` | 当前是否属于“用户手动输入的自定义分类链路” |

## 五、分类状态流转

这部分是“当前实现里的真实状态机”。

### 5.1 当前实际存在的几种状态

| 状态名 | 字段组合 | 含义 |
|---|---|---|
| 已确认系统分类 | `category_type=system` 且 `needs_category_review=false` | 已明确使用 taxonomy 中的系统分类 |
| 已确认自定义分类 | `category_type=custom` 且 `needs_category_review=false` | 已明确使用用户自定义分类 |
| 待确认推荐分类 | `category_type=pending` 且 `needs_category_review=true` | 系统已有推荐，但用户还没确认 |
| 待确认手动自定义分类 | `category_type=custom` 且 `needs_category_review=true` | 用户手动输入了自定义分类，但还没完成确认闭环 |
| 历史空态 / 旧数据未归一 | `category_type/category_key` 为空 | 旧数据或特殊回流数据，读取时会尽量迁移归一化 |

关键点：

- 当前实现里，分类状态不能只看 `category_type`
- 真实状态要看 `category_type + needs_category_review`

### 5.2 标准流转 1：手动新建，用户不选分类

```text
用户新建
-> 未明确选择分类
-> buildSavedPromptMetadata() 用启发式给一个 fallback
-> 首次落盘通常是 pending
-> asyncEnrichPrompt() 再补 AI 推荐
-> 可能自动收口为 system/custom
-> 如果仍未收口，用户手动确认
```

适用入口：

- Popup 新建
- 快速添加
- 图片转提示词保存
- 视频结果直接保存

### 5.3 标准流转 2：手动新建，用户明确选系统分类

```text
用户新建
-> 分类控件明确选择系统分类
-> 保存时带 category_source=system
-> 直接落 category_type=system
-> needs_category_review=false
-> 分类当场闭环
```

适用入口：

- Popup 新建
- Popup 编辑

### 5.4 标准流转 3：手动新建，用户输入自定义分类

```text
用户新建
-> 分类控件切到 custom
-> 保存时带 manual_custom_category=true
-> 先落 category_type=custom
-> 默认 needs_category_review=true
-> 用户后续点“确认当前分类”
-> 才变成已确认自定义分类
```

关键点：

- 当前代码里，手动输入自定义分类并不是“一保存就闭环”
- 它默认会进入“待确认手动自定义分类”

### 5.5 标准流转 4：用户选择已有自定义分类

这条是业务上应该单独存在的一类，不能和“新建自定义分类”混在一起。

```text
用户新建或编辑
-> 分类控件切到“我的分类”
-> 选择一个已经存在的自定义分类
-> 保存时带 category_source=mine
-> 业务上应直接落 category_type=custom
-> needs_category_review=false
-> 分类当场闭环
```

关键点：

- 这和“手动输入一个全新的自定义分类”不是同一种行为
- 因为这里不是在创造新分类，而是在复用一个已经存在、已经被用户认可过的分类
- 从业务语义上，它应该和“选择系统分类”一样，直接闭环

### 5.6 标准流转 5：智能转换高置信度

```text
智能转换
-> AI 直接返回 recommended_category_type/key/confidence
-> confidence >= 0.8
-> 首次保存直接落 system 或 custom
-> needs_category_review=false
-> 分类当场闭环
```

关键点：

- Smart Convert 是最短的分类链路
- 因为它在保存前就已经拿到 AI 推荐结果

### 5.7 标准流转 6：智能转换低置信度

```text
智能转换
-> AI 返回分类推荐
-> confidence < 0.8
-> 首次保存落 category_type=pending
-> needs_category_review=true
-> 用户后续选择：
   - 确认当前推荐
   - 应用 AI 推荐
   - 改成别的系统分类
   - 改成自定义分类
-> 分类闭环
```

关键点：

- Smart Convert 默认 `skip_async_enrich=true`
- 所以这类 prompt 保存后通常不会再自动跑第二次 enrich
- 低置信度状态主要靠用户自己闭环

### 5.8 标准流转 7：快速添加 / 原文保存

```text
快速添加
-> 先按原文保存
-> 初始通常是 pending
-> 后台 async enrich
-> 若 AI 给出高置信度结果，可自动收口
-> 否则继续 pending，等待用户确认
```

关键点：

- 这是“先存后补”的典型链路
- 对用户体验更稳，不会因为 AI 失败而丢数据

### 5.9 标准流转 8：编辑已有 prompt

```text
打开编辑弹窗
-> 如果分类没改，默认沿用原状态
-> 如果分类改了，则按新输入重算
-> 如果点“确认当前分类”或“应用 AI 推荐”
-> 直接把待确认状态收口
-> 更新版本历史
```

关键点：

- 编辑链路不是纯覆盖
- 它会结合 `existingPrompt` 判断用户是否真的改了分类

### 5.10 标准流转 9：导入 / 回流

```text
导入数据
-> 若已有 category_type/category_key
-> 先做 normalize
-> 若没有结构化分类
-> 重新走统一 metadata 推断
-> 必要时再 async enrich
```

适用入口：

- JSON 导入
- Obsidian frontmatter 回流

## 六、分类链路有哪几种

从“分类是怎么来的”这个角度，当前实现和业务预期合起来可以归成 7 种链路：

### 6.1 用户显式选择系统分类

最短、最稳定、最确定。

```text
用户选择系统分类
-> category_source=system
-> 直接落 system
-> 闭环完成
```

### 6.2 用户显式输入自定义分类

当前实现中不是立即闭环，而是“先记录，再确认”。

```text
用户输入自定义分类
-> manual_custom_category=true
-> 先落 custom + needs_category_review=true
-> 用户确认
-> custom + needs_category_review=false
```

### 6.3 用户选择已有自定义分类

这是你补充出来、业务上必须单列的一种链路。

```text
用户从已有自定义分类列表中选择
-> category_source=mine
-> 直接落 custom
-> 直接闭环
```

它的业务语义更接近：

- 选择系统分类

而不是：

- 新建一个新的自定义分类

### 6.4 AI 直接分类

典型入口是 Smart Convert。

```text
AI 返回推荐分类
-> 高置信度直接确认
-> 低置信度先 pending
```

### 6.5 启发式 fallback 分类

典型入口是快速添加、图片保存、视频保存。

```text
没有显式分类输入
-> 先用关键词匹配 taxonomy
-> 不够明确时给 pending
```

### 6.6 保存后异步 enrich 分类

这是“先存后补”的二段式链路。

```text
先保存
-> 后台再请求 AI
-> 补 title/tags/output_modality/category
-> 视置信度决定是否收口
```

### 6.7 外部结构化分类回流

```text
外部数据已带 category_type/category_key
-> normalize
-> 直接回流
```

## 七、哪些入口会自动闭环，哪些要用户手动闭环

### 7.1 大概率自动闭环的入口

- 用户明确选择系统分类的新建 / 编辑
- 用户选择已有自定义分类的新建 / 编辑
- Smart Convert 高置信度
- 默认 Prompt seed
- 已带结构化分类的导入 / Obsidian 回流
- 快速添加后异步 enrich 且 AI 置信度足够高

### 7.2 主要依赖用户手动闭环的入口

- Smart Convert 低置信度
- 手动输入自定义分类
- 快速添加但 enrich 仍然低置信度
- 图片保存 / 视频保存但 enrich 仍然低置信度

## 八、当前实现和设计稿的几个关键差异

这部分是后续继续重构分类时必须注意的。

### 8.1 状态不只由 `category_type` 决定

设计稿里倾向于：

- `system`
- `custom`
- `pending`

就够了。

但当前实现里真实状态至少要看：

- `category_type`
- `needs_category_review`
- `manual_custom_category`
- `ai_category_*`

另外从业务设计上，还应该额外区分两个动作：

- 选择已有自定义分类
- 输入一个全新的自定义分类

### 8.2 Smart Convert 已经允许推荐已有自定义分类

当前 prompt 和业务逻辑不是“只能推荐系统分类”，而是：

- 如果已有自定义分类明显更合适
- AI 可以直接返回 `recommended_category_type=custom`

### 8.3 手动自定义分类默认不是直接确认态

当前是：

```text
输入自定义分类
-> custom
-> needs_category_review=true
```

不是：

```text
输入自定义分类
-> custom
-> 立即确认
```

### 8.4 选择已有自定义分类，业务上应该直接闭环，但当前实现未完全体现

从业务语义上：

```text
选择已有自定义分类
-> 应直接闭环
```

因为这是在复用已有分类，不是在新建分类。

但当前实现里，`category_source=mine` 最终仍会走：

```text
custom
-> needs_category_review = !reviewConfirmed
```

也就是说：

- 代码层现在没有把“选择已有自定义分类”和“手动新建自定义分类”完全区分开
- 这正是后续分类状态机收口时应该优先修正的一点

### 8.5 视频保存传了展示分类，但没有真正传结构化分类

视频保存现在会传 `category: d.category`，但没传 `category_type/category_key`。

结果是：

- `d.category` 不是权威分类输入
- 最终仍会按统一 fallback / enrich 重算分类

### 8.6 快速添加 / 智能转换有两套并行实现

分别在：

- `background.js`
- `lib/context-menu.js`

业务上是同一类入口，但现在存在双份保存逻辑，后续最容易漂移。

## 九、后续最适合的收口方向

如果后面继续做分类收口，最值得优先统一的是这几件事：

1. 先明确“手动自定义分类保存后是否要立即闭环”
2. 明确“AI 是否允许直接推荐已有自定义分类”
3. 把快速添加和右键菜单保存逻辑收成一套
4. 把智能转换和右键智能转换保存逻辑收成一套
5. 统一视频保存时的结构化分类输入
6. 把“分类状态”定义成一份单一规则，避免 UI 和存储各自解释

---

## 十、结论

当前系统已经具备“统一 metadata -> 统一存储 -> 广播刷新 -> 异步补全 -> 分类确认”的完整闭环。

但如果只看设计稿会低估两件事：

1. 分类状态实际上比 `system/custom/pending` 更复杂
2. 不同入口的“首次分类”和“最终闭环”并不完全相同

最稳定的理解方式是：

- “保存链路”按业务入口分
- “分类状态”按字段组合分
- “分类闭环”按是否还需要 review 分

这样后续无论是继续重构分类，还是做 Hub 对接、fork、本地同步，都不会把现有状态流转搞丢。
